## T02: 平台兼容层

**目标**: 抽象 macOS/Linux 系统差异，上层模块无需关心平台细节

**产出文件**: `app/core/platform_compat.py`

**详细要求**:

1. 实现 `PlatformConfig` 类，通过 `sys.platform` 自动检测:

| 属性 | macOS (darwin) | Linux |
|:---|:---|:---|
| `shm_prefix` | `"fastvox_"` | `"fastvox_"` |
| `uds_dir` | `/tmp/fastvox/` | `/run/fastvox/` |
| `process_start_method` | `"spawn"` | `"forkserver"` |
| `recommended_workers` | `2` | `min(cpu_count, 4)` |
| `shm_cleanup_paths` | N/A (用 shm_open) | `glob("/dev/shm/fastvox_*")` |

2. 提供 `ensure_uds_dir()` 方法，自动创建 UDS 目录并设置权限 `0o755`
3. 提供 `cleanup_stale_shm()` 方法:
   - Linux: 扫描 `/dev/shm/fastvox_*` 并 unlink
   - macOS: 尝试 attach 已知 SHM name，若存在则 close + unlink
4. 提供全局单例 `platform = PlatformConfig()`

**验收标准**: 在 macOS 和 Linux 上 `import` 无报错，属性返回正确值

## T03: 共享内存管理

**目标**: 实现 Ring Buffer 模式的 SHM 管理器

**产出文件**: `app/core/shm_manager.py`

**依赖**: T02

**详细要求**:

1. 常量定义:
   ```python
   SHM_NAME = "fastvox_shm_main"
   SLOT_COUNT = 5        # 来自 Settings
   SLOT_SIZE = 2 * 1024 * 1024  # 2MB
   ```

2. `SHMManager` 类:
   - `create()`: 调用 `platform.cleanup_stale_shm()` 后创建 SHM
   - `attach()`: Worker 子进程附加已有 SHM (create=False)
   - `get_slot_view(slot_id) -> memoryview`: 返回 Slot 对应的内存视图
   - `write_to_slot(slot_id, data: bytes, offset: int = 0)`: 写入数据
   - `read_from_slot(slot_id, offset: int, size: int) -> bytes`: 读取数据
   - `destroy()`: close + unlink，捕获所有异常避免清理失败

3. 边界检查: `slot_id` 越界、数据超过 `SLOT_SIZE` 时抛出 `ValueError`

**验收标准**: 单元测试 - 主进程写入 Slot 0，子进程从 Slot 0 读出相同数据

## T05: UDS 信令协议

**目标**: 实现主进程与 Worker 之间的定长信令通信

**产出文件**: `app/core/uds_protocol.py`

**依赖**: T02

**详细要求**:

1. 信令帧格式 (16 字节):
   ```python
   SIGNAL_FORMAT = 'IIII'  # slot_id, offset, size, status
   SIGNAL_SIZE = 16
   ```

2. Status 枚举:
   ```python
   class SignalStatus(IntEnum):
       IDLE = 0
       WRITING = 1
       READY = 2
       ERROR = 3
       HEARTBEAT = 4
       COMPLETE = 5
   ```

3. `UDSServer` (主进程):
   - `start(loop)`: 创建 `AF_UNIX` + `SOCK_DGRAM` socket，绑定路径，注册 `add_reader`
   - `on_signal_callback`: 可注册的回调函数 `(slot_id, offset, size, status) -> None`
   - `stop()`: 注销 reader，关闭 socket，删除 socket 文件

4. `UDSClient` (Worker):
   - `__init__(gateway_addr)`: 连接到 Gateway 的 UDS 地址
   - `send_signal(slot_id, offset, size, status)`: pack + sendto
   - `close()`: 关闭 socket

5. 平台注意: macOS UDS path 长度限制 104 字节，路径放在 `/tmp/fastvox/`

**验收标准**: 测试 Server 收到 Client 发送的 5 个不同 status 信令

## T06: Worker 进程池

**目标**: 管理推理子进程的生命周期

**产出文件**: `app/core/worker_pool.py`

**依赖**: T02, T03, T05

**详细要求**:

1. `WorkerPool` 类:
   - `start(num_workers)`: 使用 `multiprocessing.Process` 创建 Worker
     - 设置 `start_method` 为 `platform.process_start_method`
     - 每个 Worker 传入: `worker_id`, `shm_name`, `uds_gateway_addr`, `model_dir`
   - `stop()`: 向所有 Worker 发送 SIGTERM，等待 5s，若不退出则 SIGKILL
   - `restart_worker(worker_id)`: 重启单个 Worker (崩溃恢复)
   - `health_check() -> dict`: 返回每个 Worker 的存活状态

2. Worker 间通过 `multiprocessing.Queue` 分发任务:
   ```python
   # 任务结构
   @dataclass
   class InferenceTask:
       task_id: str
       slot_id: int
       text: str
       prompt_audio_path: str | None
       prompt_text: str | None
       speed: float = 1.0
   ```

3. 每个 Worker 持有独立的 `sherpa-onnx` 引擎实例（避免共享状态）

**验收标准**: 启动 2 个 Worker，发送任务，Worker 写入 SHM 并通过 UDS 回报

## T07: TTS 引擎封装

**目标**: 封装 sherpa-onnx 的 zipvoice 模型推理

**产出文件**: `app/inference/tts_engine.py`

**依赖**: T01

**详细要求**:

1. `TTSEngine` 类:
   ```python
   class TTSEngine:
       def __init__(self, model_dir: str, num_threads: int = 1, num_steps: int = 4):
           """初始化 sherpa_onnx.OfflineTts"""
           
        def synthesize(self, text: str, speed: float = 1.0,
                       prompt_audio: str = None, prompt_text: str = None
                       ) -> tuple[bytes, int]:
            """
            执行 In-context Learning 推理。
            ZipVoice 模式下，prompt_audio 必须为参考音频文件路径，prompt_text 为对应标注。
            返回 (pcm_int16_bytes, sample_rate)
            """
           
       def warmup(self):
           """空跑一次推理消除冷启动"""
   ```

2. sherpa-onnx 配置:
   - `zipvoice.encoder` = `{model_dir}/encoder.onnx`
   - `zipvoice.decoder` = `{model_dir}/decoder.onnx` 
   - `zipvoice.vocoder` = `{model_dir}/vocos_24khz.onnx`
   - `zipvoice.tokens` = `{model_dir}/tokens.txt`
   - `zipvoice.lexicon` = `{model_dir}/lexicon.txt`
   - `zipvoice.data_dir` = `{model_dir}/espeak-ng-data`
   - `num_threads=1`, `provider="cpu"`

3. PCM 输出格式: mono, int16, sample_rate 由模型返回 (通常 24000)
4. 异常处理: 推理失败时抛出自定义 `TTSInferenceError`

**验收标准**: 调用 `synthesize("你好世界")` 返回有效 PCM 数据

## T08: Slot 调度器

**目标**: 管理 Slot 分配、排队、超时熔断

**产出文件**: `app/core/slot_manager.py`

**依赖**: T03, T06

**详细要求**:

1. `SlotManager` 类:
   ```python
   class SlotManager:
       SLOT_TTL: int = 30  # 秒
       
       async def acquire(self, request_id: str) -> int:
           """获取空闲 Slot，无空闲则 await 等待"""
           
       def release(self, slot_id: int):
           """释放 Slot 并唤醒等待队列"""
           
       async def submit_task(self, slot_id: int, task: InferenceTask):
           """将任务通过 Queue 发送给空闲 Worker"""
   ```

2. 内部使用 `asyncio.Semaphore(SLOT_COUNT)` 控制并发
3. Watchdog 后台任务: 每 5s 检查占用超时的 Slot，强制释放
4. 维护 `slot_id → {request_id, acquired_at, status}` 映射表

**验收标准**: 6 个并发请求，前 5 个立即获得 Slot，第 6 个等待

## T13: TTS 合成 API (WebSocket)

**目标**: 实现 Web 端实时流式语音合成

**产出文件**: `app/api/tts.py`

**依赖**: T08, T12

**详细要求**:

1. `WS /api/v1/tts/stream`:
   - 连接时验证 JWT (query param `?token=xxx` 或首帧)
   - 接收 JSON 请求: `{"text", "voice_id?", "speed?"}`
   - 从数据库获取对应的 `audio_path` 和 `prompt_text`
   - 调用 SlotManager 获取 Slot → 提交 InferenceTask (携带参考音频上下文)
   - 监听 UDS 信令 → 从 SHM 读取 PCM → 编码 Opus → Binary Frame 推送
   - 推理完成后发送空帧表示结束
   - 释放 Slot

2. 错误处理:
   - Slot 获取超时 → JSON Frame `{"error": "queue_full"}`
   - 推理失败 → JSON Frame `{"error": "inference_failed"}`
   - 连接断开 → 释放 Slot + 取消推理

**验收标准**: WebSocket 客户端连接后发送文本，收到可播放的 Opus 音频流

### 20.7 WebSocket 集成

1. 连接 `/api/v1/tts/stream?token=xxx`
2. 发送 JSON: `{"text", "voice_id?", "speed?"}`
3. 接收 Binary Frame → `AudioContext.decodeAudioData()` → 播放
4. 合成中状态:
   - 合成按钮变为 "合成中..." + 脉冲动画, disabled
   - 播放器区域显示 Skeleton 骨架屏
5. 完成: 收到空帧 → 恢复按钮, 播放器就绪
6. 错误: 收到 JSON `{"error": "..."}` → Toast 提示 (从顶部滑入, `300ms`)
